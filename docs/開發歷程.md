## 🔧 第二步：資料庫設置

### 問題1：資料庫表格重複建立
**現象**：執行SQL時顯示「relation "users" already exists」

**原因**：之前已經建立過表格

**解決方案**：認識到表格已存在，直接進行下一步

### 問題2：RLS安全限制
**現象**：註冊新帳號時顯示「row violates row-level security policy」

**原因**：Supabase開啟了RLS（行級安全）但沒有設置規則

**解決方案**：在Supabase中對所有表禁用RLS：
```sql
ALTER TABLE users DISABLE ROW LEVEL SECURITY;
```

### Supabase連線設定
成功取得並設置：
- Project URL
- Public API Key

填入後端`.env`檔案中，完成資料庫連接。

## 🚀 第三步：後端與前端啟動

### 問題1：jsonwebtoken版本不存在
**現象**：npm install失敗，提示「No matching version found for jsonwebtoken@^9.1.0」

**原因**：該版本號不存在於npm倉庫

**解決方案**：改用存在的版本 `^9.0.2`

### 問題2：PostCSS配置格式錯誤
**現象**：前端啟動時出現「module is not defined in ES module scope」

**原因**：postcss.config.js使用CommonJS語法，但專案設置為ES Module

**解決方案**：將檔案改名為`postcss.config.cjs`並用CommonJS語法

## 📚 第五步：教材管理功能開發

這是最複雜的部分，遇到許多挑戰。

### 問題1：檔案上傳路徑錯誤
**現象**：上傳檔案時顯示「ENOENT: no such file or directory」

**原因**：Multer的路徑設置錯誤，指向不存在的目錄

**解決方案**：
1. 手動建立 `backend/public/uploads` 目錄
2. 改用內存存儲（multer.memoryStorage）
3. 使用 fs.writeFileSync 手動寫入磁盤

### 問題2：中文檔名編碼問題
**現象**：上傳中文檔名時發生亂碼

**原因**：Windows系統與Node.js的字符編碼不相容

**解決方案**：改成用時間戳 + 隨機數生成檔名，只保留副檔名：
```javascript
const filename = uniqueSuffix + ext; // 例如：1773066057518-440821096.pdf
```

### 問題3：後端連接中斷
**現象**：前端顯示ECONNREFUSED（無法連接後端）

**原因**：後端終端機沒有保持運行

**解決方案**：
- 確保後端終端機單獨運行npm run dev
- 前端用另一個終端機運行

### 問題4：下載檔案變成HTML
**現象**：下載的檔案無法開啟，內容是HTML錯誤頁面

**原因**：Express的靜態文件服務出錯，返回了錯誤頁面

**解決方案**：建立專門的下載API端點，使用res.sendFile()並設置正確的MIME type

### 問題5：副檔名字段長度超限
**現象**：上傳時顯示「value too long for type character varying(50)」

**原因**：存儲完整MIME type（例如application/vnd.openxmlformats...）超過了50字符限制

**解決方案**：只存儲副檔名（例如「pdf」、「docx」），而不是完整MIME type

### 問題6：變數重複宣告
**現象**：後端啟動失敗，提示「Identifier 'ext' has already been declared」

**原因**：在同一個函數中宣告了兩次ext變數

**解決方案**：使用不同的變數名（extName）避免衝突

### 問題7：下載檔案格式錯誤
**現象**：
- 檔名變成「檔案名.application_pdf」
- 下載下來的檔案無法開啟

**原因**：多處邏輯出錯導致檔名和MIME type設置不正確

**解決方案**：
1. 確保ext只包含副檔名，不含點號
2. 用encodeURIComponent處理中文檔名
3. 建立MIME type對照表
4. 驗證檔案是否真的存在

### 問題8：學生端下載仍有問題
**現象**：學生頁面下載檔案時仍然是HTML或奇怪的檔名

**原因**：CourseDetail.jsx還在用舊的直接連結方式，而不是新的API

**解決方案**：更新CourseDetail.jsx改用新的下載API端點

## 🎓 主要學到的事

### 1. 檔案上傳的複雜性
想象上傳檔案看起來很簡單，但實際上涉及：
- 路徑管理
- 檔名編碼
- MIME type判斷
- 磁盤寫入驗證

### 2. 前後端協調的重要性
前端和後端改變時，另一端也要跟著更新：
- 改變後端API → 前端調用方式要改
- 改變資料格式 → 兩端都要配合

### 3. 終端多工管理
需要同時運行：
- 後端伺服器
- 前端開發伺服器
- 用不同的終端機視窗避免衝突

### 4. 增量式問題解決
每次只修復一個問題，避免改動過多導致新的bug。

## 🎯 第七步：作業系統規劃（待實現）

### 設計理念
為了讓學員有效練習與修正，採用**迭代式作業系統**：

### 核心需求

**1. 無截止日期政策**
- 作業永久開放提交
- 只要有繳交就會出現評分

**2. 多次迭代與修正**
- 學員可無限次提交修改版本
- 老師會針對最新版本重新評分
- 鼓勵學員追求完美

**3. 版本管理邏輯** ⭐ 關鍵設計
- **未批改狀態**：新上傳檔案覆蓋舊檔案
- **已批改狀態**：舊版本被鎖定（不能修正/刪除），新上傳視同「再次調整」，保留原先評分紀錄

**4. 評分等級**（與遊戲化機制銜接）
- 🔴 `建議需調整` - 需要改進
- 🟢 `合格` - 達到要求
- 🟡 `優秀` - 超出期望

**5. 批改回饋機制**
- 文字指導（老師輸入）
- 上傳範例檔案或內嵌圖片說明
- 所有回饋與評分紀錄永久保留

**6. 歷史記錄追蹤**
- 記錄每次提交時間
- 記錄提交次數
- 完整的評分歷史（展示調整歷程）

**7. 學習層次保護** ⭐ 前置解鎖機制
支援兩種模式：
- **硬性要求**：完成作業X且達到「合格」才能查看教材Y
- **建議性**：提示「建議先完成作業X」，但可跳過

教材欄位新增：
```
required_assignment_id: 關聯作業
required_grade: 需要達到的等級（建議調整/合格/優秀）
is_required: true(硬性)/false(建議性)
```

### 需要的資料庫表

1. **assignments** - 作業基本資訊
2. **submissions** - 學員提交紀錄（含版本號）
3. **feedback** - 老師批改回饋
4. 修改 **course_materials** - 加入前置解鎖欄位

### 實現優先順序
1. ✅ 資料庫設計
2. ⏳ 後端API（作業CRUD、提交、批改）
3. ⏳ 教師端UI（建立作業→查看提交→批改評分）
4. ⏳ 學生端UI（查看作業→提交→查看評分）
5. ⏳ 前置解鎖邏輯實現

## 🧪 本次真實卡關與排錯脈絡（重要交接）

### 卡關1：`查看提交` 一開始看起來是空白
**現象：**
- 老師直接點「查看提交」tab 時，頁面幾乎空白
- 但如果先從「作業列表」點進某筆作業，提交內容又能顯示

**根因：**
- 前端渲染條件綁在 `selectedAssignment`，未先選作業時整塊不渲染

**修正：**
- 進入 `submissions` tab 時，若尚未選作業：
  - 自動選第一筆作業並載入
  - 並提供 fallback 的「請先選擇作業」UI

---

### 卡關2：批改後重交看起來像「覆蓋舊紀錄」
**現象：**
- 使用者認為舊版作答與批改被蓋掉

**根因（其實是兩層問題）：**
1. 真的有邏輯風險：`is_latest` 查詢若不穩，可能選錯最新版本  
2. 顯示層誤導：教師端只看 `is_latest=true`，舊版本被隱藏，造成「被覆蓋」錯覺

**修正：**
- 提交時僅取最新一筆（排序 + limit）避免歧義
- 已批改才建立新版本；未批改覆蓋同版本（符合原需求）
- 教師端新增「版本歷程展開」可看到每版作答與每版回饋

---

### 卡關3：時區顯示錯 8 小時
**現象：**
- 上傳時間顯示下午，但實際已是晚上（差 8 小時）

**根因：**
- 資料庫欄位為 `timestamp without time zone`
- 前端/資料庫對該欄位的時區解讀不一致

**修正分兩層：**
1. **短期止血**：前端統一 `Asia/Taipei` formatter，無時區字串先按 UTC 解讀  
2. **根治**：提供 SQL migration 把主要時間欄位改為 `timestamptz`

---

### 卡關4：新增填空題作業失敗（RLS）
**現象：**
- 建立作業報錯：`new row violates row-level security policy for table "assignment_questions"`

**根因：**
- 新增資料表預設啟用 RLS，但後端使用 publishable key，無 policy 導致 insert 被拒

**修正：**
- 先提供快速修復 SQL：關閉相關表 RLS（與專案現況一致）
- 並註記上線前需改成 service role 或完整 policy

---

### 卡關5：PDF 匯出導致後端崩潰
**現象：**
- 一按 PDF 匯出，`npm run dev` 直接 Failed
- 先出現字型錯誤，接著 `ERR_STREAM_WRITE_AFTER_END`

**根因：**
1. `pdfkit` 載入某些 Windows 字型（特別是 `.ttc`）不穩  
2. 串流已開始後又進行錯誤回應寫入，導致二次 write

**修正：**
- 字型策略改為優先 `.ttf`，並加 try/catch fallback
- `doc` 串流加 error handler，避免炸整個 server
- catch 中避免對已送 header 的 response 再次寫入

---

### 卡關6：老師看到的答案像「英文+數字編碼」
**現象：**
- 老師端答案預覽出現 UUID/key，看不懂內容

**根因：**
- 前端直接顯示 `answers` 原始 JSON key-value，key 為 question id

**修正：**
- 後端先做答案解碼（題號/題名/描述/答案）
- 前端顯示人類可讀格式，並支援展開全部答案

---

### 卡關7：填空題建立後，學生端有時看不到題目輸入區
**現象：**
- 作業看起來像有填空題內容，但學生端仍顯示成檔案上傳

**根因：**
- `assignment_type` 與 `questions` 實際資料偶發不同步

**修正：**
- 後端建立作業時加「以題目數為準」的型別保險（有題目即 form）
- 前端渲染也加保險：`assignment_type=form` 或 `questions.length>0` 都視為填空題

### 阻礙與克服

**阻礙 1：回覆 API 回傳 404**
- **現象**：管理員點擊「回覆」送出後顯示「回覆失敗」或 404
- **原因**：`app.use('/api', assignmentRoutes)` 會攔截所有 `/api/*` 請求，`/api/admin/contact-messages/:id/reply` 被 assignment router 先處理，該 router 無此路由故回傳 404
- **克服**：將 messages、admin contact-messages 改為直接定義在 `app` 上（`app.post`、`app.get`、`app.patch`），不再透過 router 掛載，確保路由優先匹配

**阻礙 2：學員發送訊息後看不到、管理者也收不到**
- **現象**：發送成功（有 alert）但訊息列表為空，或訊息閃現後消失
- **原因 1**：`fetchMyMessages` 失敗時會執行 `setMyMessages([])`，覆蓋了樂觀更新
- **原因 2**：messages 路由可能被 assignment router 攔截，GET `/api/messages/me` 回傳 404
- **克服**：
  - 前端：`fetchMyMessages` 失敗時不再清空列表，保留樂觀更新
  - 後端：將 `POST /api/messages`、`GET /api/messages/me` 改為直接定義，掛載於 assignment router 之前
  - 加入樂觀更新：發送成功後立即顯示新訊息，再背景重新取得

**阻礙 3：登入失敗、無法連線**
- **現象**：重啟後端時一併關閉了所有 node 程序，前端 dev server 也被關閉
- **原因**：前端未運行時，`/api` 請求無法透過 Vite proxy 轉發至後端
- **克服**：重啟時需同時啟動後端與前端；Login 頁面增加「無法連線，請確認前後端已啟動」的錯誤提示

**阻礙 4：訊息送出後消失**
- **現象**：發送成功後訊息短暫出現又消失
- **原因**：`fetchMyMessages()` 背景刷新失敗時，catch 區塊執行 `setMyMessages([])` 清空了剛加入的樂觀更新
- **克服**：移除 `fetchMyMessages` 失敗時的 `setMyMessages([])`，失敗時不覆蓋既有列表

- `backend/src/index.js` - 直接定義 messages、admin contact-messages 路由
- `frontend/src/pages/Dashboard.jsx` - 樂觀更新、fetchMyMessages 錯誤處理

### 遇到的問題與解決方案

#### 問題 1：`/api/health` 一直 loading，無回應

**現象**：瀏覽器訪問 `http://localhost:3000/api/health` 持續 loading，無回傳。

**檢測結果**：
- 後端 3001 直接訪問可正常回傳 `{ ok: true }`
- 透過 Vite proxy 訪問 3000 時連線失敗或卡住
- 參考 [Vite #7620](https://github.com/vitejs/vite/discussions/7620)：Node 17+ 將 `localhost` 解析為 IPv6 (`::1`)，可能導致 proxy 連線異常

**解決方案**：
1. `frontend/vite.config.js`：proxy `target` 改為 `http://127.0.0.1:3001`（強制 IPv4）
2. 新增 `timeout: 5000`，避免 proxy 無限 pending
3. 新增 `configure` error handler：後端未啟動時回傳 502 與明確錯誤訊息
4. 新增 `server.host: '127.0.0.1'`，讓 Vite 綁定 IPv4，確保 127.0.0.1 可連線

**修改檔案**：`frontend/vite.config.js`

---

#### 問題 2：後端失聯，重啟失敗

**現象**：後端斷線後重啟，出現 `Error: listen EADDRINUSE: address already in use :::3001`。

**檢測結果**：port 3001 被舊的 node 行程佔用。

**解決方案**：
```powershell
Get-NetTCPConnection -LocalPort 3001 | ForEach-Object { Stop-Process -Id $_.OwningProcess -Force }
```
終止佔用行程後重新啟動 backend。

---

#### 問題 3：前後端都連不進去

**現象**：清理後重啟，後端 3001 正常，但前端 3000 仍無法連線。

**檢測結果**：Vite 預設可能只綁定 IPv6，用 127.0.0.1 連線時被拒絕。

**解決方案**：在 `vite.config.js` 加上 `server.host: '127.0.0.1'`，強制綁定 IPv4。

---

#### 問題 4：特等勳章第 1 次優良就觸發

**現象**：學員第一次獲得「優秀」評級時，特等勳章（應為累計 5 次）就觸發。

**根因**：`checkAndAwardBadges('excellent_5')` 只檢查是否已有該勳章，未驗證優秀次數。

**解決方案**：
- 新增 `getExcellentGradeCount(userId)`：查詢 submissions（is_latest=true）與 feedback，取每筆提交最新 grade，統計 `grade='優秀'` 的筆數
- `onExcellentGrade` 改為：`excellentCount >= 1` 才頒優良勳章，`excellentCount >= 5` 才頒特等勳章

**修改檔案**：`backend/src/badgeService.js`

---

#### 問題 5：六藝勳章第 7 個才觸發

**現象**：學員獲得第 6 個勳章時未觸發六藝勳章，第 7 個才觸發。

**根因**：`checkBadgesCount6` 條件為 `count < 6`，需已有 6 個勳章才頒六藝，導致六藝成為第 7 個。

**解決方案**：將條件改為 `count < 5`。當已有 5 個勳章時頒發六藝，六藝即為第 6 個勳章。

**修改檔案**：`backend/src/badgeService.js`

---

#### 問題 6：管理員重複頒發勳章仍加經驗值

**現象**：學員已擁有某勳章，管理員再次頒發時仍會增加經驗值。

**根因**：`awardBadge` 使用 `upsert`，未先檢查是否已擁有。

**解決方案**：
1. 頒發前先查詢 `user_badges` 是否已有該勳章
2. 若已擁有，回傳 `{ awarded: false, alreadyHas: true }`，不寫入、不加經驗
3. 改為 `insert` 取代 `upsert`，避免覆蓋
4. API 回傳 400：`該學員已獲得此勳章，不會重複頒贈，也不會獲得經驗值`

**修改檔案**：`backend/src/badgeService.js`、`backend/src/index.js`

## 2026-03-14 優秀作品填空題觀摩修正

### 開發內容

**問題**：填空題優秀作品在優秀作品區，學員無法觀摩到填答者的答案；上傳檔案類型則無此問題。

**根因**：
1. `GET /courses/:courseId/featured-submissions` 僅回傳 submissions，未包含 form_submissions 的 `resolved_answers`
2. `GET /submissions/:submissionId` 對學員限制過嚴：僅能查看自己的提交，觀摩他人優秀作品時回傳 403

**解決方案**：
1. **featured-submissions API**：對 `assignment_type === 'form'` 的提交，查詢 form_submissions 並呼叫 `resolveFormAnswers`，將 `resolved_answers` 一併回傳
2. **submissions/:id API**：學員可查看自己的提交，或 `is_featured === true` 的他人作品（觀摩用），並正確回傳填空題的 `resolved_answers`

**修改檔案**：`backend/src/assignments.js`

### 困難一：Google Drive 上傳失敗（storageQuotaExceeded）

**現象**：檔案上傳顯示成功、可下載，但 Google Drive 中看不到檔案

**原因**：Service Account 沒有儲存配額，無法在「一般資料夾」建立檔案。錯誤訊息：`Service Accounts do not have storage quota. Leverage shared drives...`

**嘗試方法**：
- 最初使用一般資料夾並共用給 Service Account → 失敗
- 查詢後發現必須使用「共用雲端硬碟」（Shared Drives）

**解決方案**：
- 改用 **共用雲端硬碟**（僅 Google Workspace 帳號可建立）
- 使用者以另一有 Workspace 的帳號建立共用雲端硬碟，加入 Service Account 為「內容管理員」
- 取得共用雲端硬碟 ID，更新 Render 的 `GOOGLE_DRIVE_FOLDER_ID`
- 更新 `docs/Google_Drive_設定教學.md`，改為共用雲端硬碟流程

**結果**：✅ 上傳成功，檔案正確出現在共用雲端硬碟

---

### 困難二：Render 部署失敗（Port scan timeout）

**現象**：部署顯示 `Port scan timeout reached, no open ports detected`

**原因**：後端只綁定 `localhost`，Render 從外部無法偵測

**解決方案**：改為 `app.listen(PORT, '0.0.0.0', ...)` 綁定所有介面

**結果**：✅ 部署成功

---

### 困難三：教材下載按鈕不顯示

**現象**：檔案已上傳至 Drive，但教材頁面沒有「下載檔案」按鈕

**原因**：前端只檢查 `material.file_url`，使用 Drive 時 `file_url` 為 null，僅有 `drive_file_id`

**解決方案**：`CourseDetail.jsx`、`ManageMaterials.jsx` 改為 `(material.file_url || material.drive_file_id)` 才顯示下載按鈕

**結果**：✅ 下載功能正常

---

### 困難四：通知信未寄出

**現象**：`/api/email-status` 顯示 `emailEnabled: true`，但教材上線、批改完成時未收到信

**嘗試方法**：
- 新增 `/api/email-test?to=信箱` 測試端點
- 重新設定 Gmail 應用程式密碼

**發現**：測試端點回傳 `Connection timeout`

**原因**：**Render 免費方案自 2025/9 起封鎖對外 SMTP 連線**（埠 25、465、587），導致無法寄信

**解決方案**：
- 方案 A：升級 Render 付費方案（可正常使用 SMTP）
- 方案 B：改用 Resend、SendGrid 等 HTTP API 寄信服務
- 使用者選擇：**隱藏通知偏好 UI**，因目前無法寄信，設定無實質影響

**結果**：✅ 移除 Dashboard 個人檔案頁的「Email 通知偏好」區塊

### 困難五：部署語法錯誤（Unexpected token 'catch'）

**現象**：部署失敗，`SyntaxError: Unexpected token 'catch'`

**原因**：`emailService.js` 中 `sendSubmissionGradedEmail`、`sendCourseEnrolledEmail` 有 `catch` 區塊但缺少對應的 `try`

**解決方案**：補上 `try {` 區塊

**結果**：✅ 部署成功
