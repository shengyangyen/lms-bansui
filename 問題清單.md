## 🔴 高優先級問題

### 問題 1: 測試帳號無法登入
**描述**: 測試腳本使用的帳號不存在或密碼錯誤
- admin@example.com → 使用者不存在  
- test@example.com → 密碼錯誤
- st@example.com → 密碼錯誤

**影響**: 25/32 測試因無法取得 token 而失敗

**對用戶的影響**: 無法執行完整的自動化測試

**重現步驟**:
1. 執行 `node test-automation.js`
2. 所有登入測試失敗
3. 後續需要認證的測試全部失敗

**優先級**: 🔴 高

**建議修復**:
- 建立標準測試帳號(email + password)
- 或更新測試腳本使用正確的帳號

## 🟡 中優先級問題

### 問題 2: API 路由不匹配
**描述**: 測試腳本使用的 API 路由與後端實際路由不一致

**錯誤的路由**:
- `POST /api/assignments` → 應為 `POST /api/courses/:courseId/assignments`
- `POST /api/assignments/:id/submit-form` → 應為 `POST /api/assignments/:id/form-submit`
- `GET /api/assignments/:id/submissions/student/:studentId` → 應為 `GET /api/assignments/:assignmentId/students/:studentId/submissions-history`

**影響**: 5 個測試返回 404 錯誤

**對用戶的影響**: 作業系統功能無法完整測試

**優先級**: 🟡 中

**建議修復**: 更新測試腳本以使用正確的 API 路由

---

## 🟢 低優先級問題

### 問題 3: Token 驗證邊界條件
**描述**: 使用無效 token 時,系統回應不一致

**測試場景**: 使用明顯無效的 token (`Bearer invalid_token_12345`)

**預期**: 返回 403 錯誤  
**實際**: 測試顯示系統接受了無效 token

**對用戶的影響**: 可能存在輕微的安全隱患

**優先級**: 🟢 低

**建議修復**: 進一步驗證 JWT 中間件的錯誤處理邏輯

---

### 問題 4: UUID 參數驗證不足
**描述**: 當 UUID 參數為 null 時,PostgreSQL 返回內部錯誤

**錯誤訊息**: `invalid input syntax for type uuid: "null"`

**預期**: API 應在路由層驗證參數,返回 400 錯誤  
**實際**: 錯誤傳遞到資料庫層,返回 500 錯誤

**對用戶的影響**: 錯誤訊息不夠友善

**優先級**: 🟢 低

**建議修復**: 在路由層加入參數驗證中間件
